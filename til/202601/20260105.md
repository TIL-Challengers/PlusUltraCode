### GOF 디자인 패턴 - Decorator 패턴과 추상 팩터리 메서드, 추상 팩터리 패턴

오늘 배운건 decorator 패턴과 추상 팩터리 패턴이다.

decorator 패턴을 간단하게 설명하면 아래와 같다.
1. 추상클래스인 Beverage인 음료가 있다.
2. 해당 클래스를 상속받고 있는 여러 음료 종류들이 있다.

3. 여기가 중요한데 데코레이터 클래스를 만든다. 예를들어 DesignBeverage 형태로 용지를 채워주는 클래스? 아니면 ComponentBeverage 처럼 첨가물 들어가는 데코레이터 클래스를 만든다.
4. 데코레이터 클래스는 아래와 같이 작성한다.

```
abstract class DesignBeverage extends Beverage

Beverage beverage;

기타 메서드

```
이런식으로 멤버 필드에 beverage를 composition 해놓고 Beverage또한 확장한다. 
멤버 필드에 beverage를 놓는 이유는 상위 모듈에서 beverage = new PuppyDesing(beverage) 형태로 갱신해준다.
이전 beverage를 PuppyDesign()에 넣어서 새로운 beverage를 탄생시키기 위해서다. 
이 부분이 진짜 핵심 키 포인트 였다. 재미난 패턴이다.

### 추상 팩터리 패턴
이거는 쉽게말해서 내가 무의식적으로 쓰던 패턴이다. 
간단하게 객체를 만드는 클래스를 따로 정의하여 store라는 클래스 안에서 PizzaFactory 형태로 멤버 필드를 선언한 뒤 pizzFactory.create() 형태로 사용한다.

근데 이러면 추후 여러개의 store생길 시 확장이 다소 힘든점이 있다.

이렇게 될 경우 
store를 abstract로 선언한 뒤 PizzaFactory를 제거하고 pizzaFactory.create() 대신 create()라는 추상메서드 를 선언한다. 

이게 키포인트다. 

추상 메서드를 선언하였기 때문에 store를 상속받는 구체 클래스에서 해당 부분을 구현해야 한다. 
이렇게 작성하면 
상위 모듈에서 
```
PizzaStore pizzastore = new SuPizzaStore() 형태로 객체를 생성한뒤
pizzastore.order("cheese") 형태로 구현하면 된다.
 
```

한가지 정의하면 추상 팩터리와 vs 추상 메서드 팩터리 차이가 멀까?
추상 메서드 팩터리는 해당 클래스를 추상클래스로 만든 뒤 메서드를 추상메서드를 추가하여 구체 클래스의 역할을 위임하는 형태다.

추상 팩터리란 말 그대로 객체를 이용하여 해당 클래스에서 작업하는 형태다. 멤버필드로 주입받아서
